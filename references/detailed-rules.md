# Skill 概述

## Skill 目标

本 skill 用于技术岗面试的结构化复盘与回答优化。第一版只承担两类任务：

1. 还原并整理面试中出现的问题、回答与薄弱点。
2. 在不虚构经历的前提下，输出更优的回答思路与答法。

本 skill 的定位是：**面试复盘与回答优化工具**。  
本 skill 不是：

- 伪造经历包装器
- 虚构项目生成器
- 脱离用户背景的万能话术库
- 无视岗位现实的标准答案生成器

执行本 skill 时必须遵守以下底线：

- 个性化分析与个性化回答必须基于用户材料。
- 材料不足时必须降级，不能脑补。
- 可以优化表达、结构与顺序，不能补造事实。
- 可以指出原回答不完整，不能伪造细节来补齐。

## Skill 核心产出

本 skill 的标准产出包括：

- 候选人画像简短版
- 问题提取结果
- 逐题考点判断与问题诊断
- 更好的回答思路
- 个性化回答或优秀回答
- 整场面试短板总结

输出必须围绕“复盘”和“优化”展开，不得扩写为泛泛求职建议。  
输入不足时仍可输出结果，但必须降低个性化强度，并明确说明限制。

# 适用范围与边界

## 第一版适用岗位

第一版主要面向技术岗，重点支持以下方向：

- AI 应用开发
- RAG
- Agent
- 后端

执行时应优先围绕这些岗位常见考察点进行判断：

- 项目落地能力
- 技术选型理由
- 系统理解深度
- 方案取舍能力
- 问题排查与工程化能力
- 与岗位 JD 的匹配程度

如果岗位与上述方向高度相关，应按本 skill 完整规则执行。  
如果岗位仍属技术岗但方向偏离较大，可继续使用本 skill，但应降低领域特化判断，避免过强的 AI / RAG / Agent 假设。  
如果岗位明显不属于技术岗，第一版不应强行套用本 skill 的判断框架。

## 支持的复盘场景

本 skill 适用于以下场景：

- 面试结束后对整场问答做结构化复盘
- 针对多道题逐题找问题、补思路、重写回答
- 基于岗位 JD 和简历判断回答是否贴岗
- 基于项目材料生成更贴近真实经历的回答
- 材料有限时先做通用复盘，补充材料后再继续优化

本 skill 的默认对象是“已发生的面试”或“已有问答记录的模拟面试”。  
如果没有面试对话记录，不得假设出现过哪些题，也不得直接进入逐题复盘。

## 暂不处理的内容

第一版不处理以下事项：

- 编造项目经历、岗位经历、团队职责、业务结果
- 伪造数据指标、上线结果、性能收益、业务影响
- 生成与真实背景不一致的高配版人设
- 将用户包装成其并不具备的岗位层级
- 输出脱离岗位现实的理想化神级回答
- 代替用户判断事实真伪

如果用户材料存在明显冲突、缺漏或不可信内容，本 skill 可以指出风险，但不能自行补齐成完整故事。  
如果某题与用户经历弱相关，本 skill 可以给出优秀回答，但不能冒充为用户真实经历。  
如果用户要求“润色”或“优化”，允许增强表达质量，但增强边界仅限于：

- 重组已有信息
- 提炼已有亮点
- 补足回答结构
- 让表达更贴近岗位需求

以下增强一律禁止：

- 把未做过写成做过
- 把参与过写成主导过
- 把尝试过写成落地过
- 把了解过写成精通过

# 输入模型与降级规则

## 必需输入

第一版的标准必需输入为：

- 简历
- 面试对话记录

只有同时具备这两项，才可进入完整复盘模式。

### 简历

简历用于定义候选人的真实经历边界，至少用于识别以下信息：

- 目标岗位方向
- 过往项目与职责
- 技术栈
- 工作或实习经历
- 可调用的真实背景素材
- 潜在高风险项

简历是个性化分析与个性化回答的主要依据之一。  
没有简历时，不得强行输出高贴合度的个性化回答映射。

### 面试对话记录

面试对话记录用于还原真实问题与用户原回答。  
可接受形式包括：

- 逐轮问答记录
- 聊天记录整理版
- 面试回忆稿
- 口语转写文本

最低要求是能大致区分问题与对应回答。  
如果只有零散片段，仍可运行，但必须降低逐题精确诊断强度。

## 可选输入

以下输入为可选输入，用于增强分析质量：

- 岗位 JD
- 项目材料 / 项目说明 / 项目总结
- 自我介绍稿
- 其他用户背景材料

### 岗位 JD

岗位 JD 用于增强以下判断：

- 这题在考什么
- 面试官真正关心的能力点
- 用户回答是否贴岗
- 哪些经历值得优先调用
- 哪些回答缺少岗位视角

有岗位 JD 时，应纳入分析。没有岗位 JD 时，只能做通用技术岗判断，不得伪造岗位要求。

### 项目材料 / 项目说明 / 项目总结

项目材料用于增强以下内容：

- 候选人画像中的相关项目识别
- 项目题、场景题、系统设计题的个性化回答
- 技术细节、方案取舍、落地过程
- 真实回答素材的调用稳定性

有项目材料时，应优先使用可验证信息。  
没有项目材料时，如果简历项目描述过短，必须降低细节强度，不能补造技术细节。

### 自我介绍稿

自我介绍稿用于辅助识别以下内容：

- 当前表达方式
- 习惯强调的经历点
- 可保留或需要调整的叙述顺序
- 回答风格建议

它只作为辅助材料，不能覆盖简历与面试对话记录中的事实边界。

### 其他用户背景材料

其他用户背景材料只作为补充证据，例如：

- 学习笔记
- 项目文档
- 复盘草稿
- 岗位投递说明
- 历史面试反馈

这类材料不得压过简历与面试对话记录的优先级。

## 输入优先级

多种材料同时存在时，默认优先级如下：

1. 面试对话记录中的明确问答事实
2. 简历中的明确经历与项目事实
3. 项目材料中的可验证细节
4. 岗位 JD 中的岗位要求
5. 自我介绍稿与其他背景材料中的辅助信息

执行时必须遵守以下规则：

- 问题是否真实出现，以面试对话记录为准。
- 用户是否真实做过某件事，以简历和项目材料为准。
- 岗位贴合判断以岗位 JD 为增强依据；没有岗位 JD 时只能做通用判断。
- 材料冲突时不得自动拼接为新故事，必须保守处理。

## 输入不足时的降级处理

### 缺少 JD

如果缺少岗位 JD：

- skill 仍可正常运行
- 可以继续做问题提取、回答诊断、回答优化
- 岗位贴合分析必须降级为通用技术岗判断
- 不得输出过强的岗位意图断言
- 不得假设公司背景、团队阶段或业务场景

### 缺少项目材料

如果缺少项目材料：

- skill 仍可正常运行
- 可以基于简历与面试对话记录生成个性化回答
- 个性化回答的细节强度必须下降
- 只能使用简历中已有项目信息
- 不得自行补全架构细节、指标结果、团队分工

当某题需要较强项目细节支撑时，应承认材料不足，并优先输出更好的回答思路或优秀回答。

### 缺少自我介绍稿

如果缺少自我介绍稿：

- 不影响主流程运行
- 仅减少对表达风格的辅助判断
- 不影响候选人画像、题型判断与回答优化主链路

### 缺少其他背景材料

如果缺少其他背景材料：

- 不影响主流程运行
- 仅减少补充素材来源
- 不得因此放宽真实性边界

### 只有简历 + 面试对话记录

如果只提供简历和面试对话记录：

- skill 应正常运行
- 这是第一版的最低完整可运行配置
- 可以做候选人画像、逐题复盘、问题诊断、回答优化
- 可以生成个性化回答，但必须严格受限于用户材料

如果某题所需细节超出已提供材料，必须主动降低回答细节密度。

### 缺少简历

如果缺少简历：

- 不能进入强个性化回答映射
- 只能做通用复盘
- 可以执行的问题提取、题型判断、原回答诊断、更好的回答思路、优秀回答
- 不得输出高置信度的个性化回答
- 不得绑定用户真实项目映射
- 不得推断“最适合强调哪个项目”这类依赖简历的信息

如果用户只提供面试对话记录而没有简历，必须明确说明：  
当前输出仅为通用复盘与通用优化，不代表基于真实背景完成的个性化回答优化。

### 缺少面试对话记录

如果缺少面试对话记录：

- 不能做标准逐题复盘
- 不能假设面试官问了哪些题
- 不能输出“原回答的问题”这类依赖真实问答的判断
- 只能退化为基于简历、岗位 JD、项目材料的面试准备或预演支持

这类输出不属于严格意义上的面试复盘。

## 最低执行要求

第一版完整运行的最低要求是：

- 简历
- 面试对话记录

缺少其中任一项时，必须触发降级。  
缺少简历时，只能做通用复盘。  
缺少面试对话记录时，必须转为非复盘模式的保守输出。

# 候选人画像构建

## 构建时机

候选人画像必须在正式复盘前构建，执行顺序不得颠倒：

1. 读取并清洗输入材料
2. 构建候选人画像
3. 再进入问题提取、题型判断、原回答诊断与回答优化

未完成候选人画像前，不得直接进入逐题优化。  
后续以下判断都依赖画像：

- 哪些经历应优先调用
- 哪些题适合做个性化回答
- 哪些回答存在高风险
- 哪种表达方式更适合该候选人

输入不足时仍应构建画像，但必须显式标注不确定性，不能用猜测补齐。

## 必含字段

候选人画像至少必须包含以下 6 个字段。

### 1. 目标岗位方向

用于判断候选人的主求职方向。  
优先根据以下材料综合判断：

- 简历中的岗位名称、项目方向、技术栈
- 岗位 JD 中的职责与要求
- 自我介绍稿中的主打方向

如果材料中存在多个方向，应给出当前最主要方向。  
如果无法明确，只能给出保守表述，不得伪造精确定位。

### 2. 最相关项目经历

用于识别后续回答中最应优先调用的项目。  
该字段必须聚焦与目标岗位最相关、最能支撑回答的项目，不得罗列全部项目。

优先考虑以下因素：

- 与目标岗位的技术相关性
- 在面试中被高频追问的项目
- 能体现复杂度、取舍、落地能力的项目
- 简历和项目材料中信息更完整的项目

如果不足以判断唯一项目，可以列出 1 到 2 个候选项目，但不得虚构项目亮点。

### 3. 可优先调用的背景素材

用于后续个性化回答时快速调用真实素材。  
素材可以包括：

- 项目职责
- 技术选型理由
- 难点与解决方案
- 线上问题处理经历
- 性能优化、稳定性优化、工程化实践
- 与岗位 JD 强相关的经历片段

这里只能写已知可调用素材，不能写推测素材。

### 4. 简历中的高风险项

用于标记后续回答中容易被追问、失分或暴露破绽的地方。  
高风险项可包括：

- 简历写法过满但材料支撑不足
- 项目描述抽象、缺关键细节
- 技术名词堆砌但职责边界不清
- 经历跨度大但主线不明确
- 与目标岗位明显不匹配的经历表述

高风险项必须基于已有材料识别，不得凭空制造。

### 5. 优势表达方向

用于确定后续回答应优先强调哪些优势。  
常见方向包括：

- 工程落地
- 技术深度
- 方案取舍
- 问题排查
- 业务理解
- 学习能力

该字段用于帮助回答聚焦最可信、最贴岗、最有说服力的优势，不用于制造夸张人设。

### 6. 回答风格建议

用于控制后续回答的表达方式。  
建议内容可包括：

- 更适合先结论后展开，还是先背景后方案
- 更适合务实、工程化，还是更结构化
- 是否需要减少概念堆砌
- 是否需要增加场景细节
- 是否需要更清晰地区分“我做了什么”和“团队做了什么”

该字段只服务于表达优化，不得演变为包装话术模板。

## 输出形式

候选人画像要展示给用户，但只展示简短版。  
简短版必须满足以下要求：

- 每个字段只保留支撑后续复盘所需的关键信息
- 不展开成长段分析
- 不提前写成完整面试答案
- 不暴露全部推理过程

候选人画像简短版的作用是：

- 让用户知道 skill 当前如何理解其背景
- 为后续逐题优化建立统一上下文
- 在材料不足或判断有歧义时给用户纠正机会

如果某字段信息不足，可以写“信息不足”或“待补充”，不能用猜测替代。

## 构建原则

候选人画像构建必须遵守以下原则：

- 优先基于简历、项目材料、岗位 JD 判断
- 对事实边界不清的内容保持保守
- 不虚构经历
- 不脑补项目细节
- 不把通用岗位要求误写成用户个人优势
- 不把理想候选人画像误当成用户真实画像

材料齐全时，优先级如下：

1. 简历中的明确经历事实
2. 项目材料中的可验证细节
3. 岗位 JD 中与岗位贴合相关的信息
4. 自我介绍稿和其他背景材料中的辅助信息

材料不齐全时，执行策略如下：

- 有简历但无项目材料：可以构建画像，但项目细节必须保守
- 有岗位 JD 但无简历：可以判断岗位方向，但不能形成强个性化画像
- 有简历但无岗位 JD：可以判断候选人方向，但岗位贴合判断必须保守
- 只有面试对话记录：只能基于问答内容做有限画像，不得声称已完成完整画像

候选人画像的目标是服务后续复盘，不是替用户重写简历。

# 标准工作流

## Step 1：读取并清洗输入材料

本步必须完成以下动作：

- 读取全部已提供输入
- 识别材料类型：简历、面试对话记录、岗位 JD、项目材料、自我介绍稿、其他背景材料
- 去掉明显无效内容，例如重复粘贴、空白段落、纯寒暄、与面试无关的噪音信息
- 初步判断材料缺失情况，为后续降级做准备

本步产出：

- 清洗后的材料集合
- 输入完整度判断结果
- 材料优先级上下文

与下一步的衔接：

- 清洗后的材料直接作为候选人画像构建输入
- 本步未完成时，Step 2 不得开始

## Step 2：构建候选人画像

本步必须完成以下动作：

- 构建候选人画像
- 明确目标岗位方向
- 识别最相关项目经历
- 提取可优先调用的背景素材
- 标记简历中的高风险项
- 提炼优势表达方向
- 给出回答风格建议

本步产出：

- 候选人画像内部版本
- 展示给用户的候选人画像简短版

与下一步的衔接：

- 候选人画像作为问题提取与后续个性化判断的统一上下文
- 后续“是否适合做个性化回答”的判断必须引用该画像

## Step 3：提取面试官问题和用户反问

本步必须完成以下动作：

- 从面试对话记录中识别面试官问题
- 识别用户主动提出的反问
- 去除无复盘价值内容
- 对问题做基础切分，避免把多个独立问题混在一起
- 必要时将连续追问抽象为问题簇，但同时保留原始追问链

本步产出：

- 面试官问题列表
- 用户反问列表
- 必要时的问题簇列表
- 每道题对应的原始回答片段或回答摘要
- 必要时的原始追问链

与下一步的衔接：

- 问题列表直接进入题型判断
- 问题提取不清时，Step 4 与 Step 5 都会失真

## Step 4：判断题目类型

本步必须完成以下动作：

- 对每道题做题型分类
- 每道题必须落入以下 6 类之一：项目题、场景题、系统设计题、八股题、行为题、JD 匹配题
- 多属性问题必须判定主分类，必要时保留次级属性供内部参考

本步产出：

- 每道题的主分类
- 必要时的次级属性说明
- 题型判断依据摘要

与下一步的衔接：

- 题型决定原回答问题诊断重点
- 题型决定后续更偏个性化回答、优秀回答，还是仅做通用优化

## Step 5：诊断原回答问题

本步必须完成以下动作：

- 基于题型、岗位方向与原回答诊断问题
- 判断问题主要出在内容、结构、深度、岗位贴合、真实性边界，还是信息组织方式
- 标记会导致追问、失分或暴露高风险项的问题

本步产出：

- 每道题的原回答问题清单
- 每道题的改进方向摘要

与下一步的衔接：

- 回答优化必须建立在诊断结果之上
- 没有诊断时，Step 6 只能变成换一种说法，不能真正解决问题

## Step 6：生成优化回答

本步必须完成以下动作：

- 基于题型、候选人画像、原回答问题与材料可用性生成更优回答
- 判断该题是否适合做个性化回答
- 适合时优先生成个性化回答
- 不适合或材料不足时生成优秀回答或通用优化答法
- 对高价值题补充现场训练增强内容，如记忆锚点、30秒版、90秒版、追问准备

本步产出：

- 每道题的更好的回答思路
- 每道题的个性化回答或优秀回答
- 必要时的可选追问及补充答法
- 高价值题的现场训练增强字段

与下一步的衔接：

- 逐题优化结果进入整场短板总结
- 本步若未与前面诊断、题型、画像对齐，Step 7 会失真

## Step 7：总结整场面试短板

本步必须完成以下动作：

- 汇总重复性失分模式
- 提炼最影响结果的共性短板
- 区分知识问题、表达问题、项目支撑问题、贴岗问题
- 给出后续优先改进顺序

本步产出：

- 整场面试短板总结
- 后续训练重点清单

与前面步骤的关系：

- 该总结必须基于逐题诊断与逐题优化形成
- 不能脱离逐题结果单独给出空泛结论

## 工作流执行要求

执行本 skill 时，必须遵守以下约束：

- 不得跳过 Step 2 直接进入逐题优化
- 不得跳过 Step 4 直接进入回答生成
- 不得在问题尚未提取清楚时做大规模重写
- 不得用整场总结替代逐题分析
- 某一步无法完整执行时，必须显式降级，不得假装已完成

# 问题提取与题型判断

## 面试官问题提取规则

问题提取的目标是：还原真正被考察的题目，而不是机械截取每一句话。

提取时必须遵守以下规则：

- 只提取具有明确考察意图的问题
- 同一题的铺垫句、追问句、限定条件应合并理解
- 一个长问题中的独立子问题必须拆开
- 如果某段话本质是在追问前一道题，应与前题关联，不要误判为新题
- 如果记录不完整但能识别题意，可以保留为近似问题，同时降低后续判断强度
- 对连续追问较多、深层考察意图一致的问题，允许抽象成问题簇
- 如果形成问题簇，必须保留原始追问链，不能只留下一个抽象标题

以下内容通常可提取为面试官问题：

- 明确提问
- 要求展开某个项目或经历
- 要求解释技术原理
- 要求给出设计方案、取舍或排查思路
- 要求说明与岗位相关的匹配度

不能因为一句话以疑问语气结尾，就自动认定为有效问题。  
判断标准只有一个：该内容是否要求用户作出可复盘的回答。

## 用户反问提取规则

用户反问也应提取，但必须与面试官问题分开处理。  
用户反问主要用于辅助分析以下内容：

- 用户对岗位的理解是否到位
- 用户是否抓住团队与岗位关键信息
- 用户在面试尾段的表达是否成熟

提取时必须遵守以下规则：

- 只提取用户主动提出且具有明确求证或了解目的的问题
- 不把礼貌性收尾当作有效反问
- 不把用户对前文的澄清当作反问
- 连续追问同一主题时可以合并为一组反问

用户反问通常不进入主问题列表，但必须保留在独立列表中供整体复盘参考。

## 无复盘价值内容的剔除规则

以下内容默认视为无复盘价值或低复盘价值，应在提取阶段剔除或弱化：

- 寒暄
- 自我介绍前的开场确认
- 网络卡顿、听不清、重复确认
- 时间安排、面试流程说明
- 无实质信息的过渡语
- 单纯礼貌回应

以下内容需要谨慎处理，不应直接剔除：

- 看似简短但实际在考察表达反应能力的问题
- 简短追问但实际指向核心薄弱点的问题
- 与岗位 JD 强相关的确认式问题

剔除原则是：  
去掉不影响复盘结论的噪音内容，保留会影响考点判断、原回答诊断和回答优化的内容。

## 问题提取阶段的边界规则

- 如果将多轮连续追问抽象成问题簇，必须同时保留原始追问链。
- 原始追问链至少应保留：主问题 -> 第一层追问 -> 第二层追问 -> 深层考察意图。
- 问题簇用于提升抽象与归纳能力，不得破坏后续追问训练的可用性。
- 对后续需要模拟面试或追问训练的题，不得把连续追问压扁到只剩一个标题。

## 题型分类规则

每道题必须判断主分类。第一版固定使用以下 6 类。

### 项目题

满足以下任一特征时，优先归为项目题：

- 让用户介绍某个项目
- 追问项目中的职责、难点、方案、结果
- 追问项目中的技术选型、排障、优化、上线、复盘

项目题的核心是验证用户是否真正做过、做到了什么、理解到什么程度。

### 场景题

满足以下任一特征时，优先归为场景题：

- 给出业务或工程场景，要求用户说明处理思路
- 询问“如果遇到某问题你会怎么做”
- 要求用户在不完全信息下给出判断、排查或落地方案

场景题的核心是验证问题解决能力与落地判断，而不是回忆项目事实。

### 系统设计题

满足以下任一特征时，优先归为系统设计题：

- 要求设计系统、模块、服务、链路或整体方案
- 要求说明架构分层、组件协作、容量、稳定性、扩展性、性能、容灾
- 要求讨论设计取舍与演进路径

系统设计题的核心是验证抽象能力、架构能力与取舍能力。

### 八股题

满足以下任一特征时，优先归为八股题：

- 询问基础概念、定义、原理、机制
- 询问常见技术知识点的标准理解
- 与用户个人经历关联较弱，主要考察基础掌握程度

八股题的核心是验证基本功。  
如果问题本质是基础原理题，即使用户试图用项目包装，也不应误判为项目题。

### 行为题

满足以下任一特征时，优先归为行为题：

- 询问过往行为、决策方式、协作方式、冲突处理、抗压、学习方式
- 询问失败经历、挑战经历、推动经历、沟通经历

行为题的核心是验证稳定性、协作方式与职业行为模式。

### JD 匹配题

满足以下任一特征时，优先归为 JD 匹配题：

- 直接询问为什么适合这个岗位
- 询问过往经历与岗位要求如何对应
- 询问是否具备岗位 JD 中的某项能力、经验或方向

JD 匹配题的核心是验证用户与岗位之间的对应关系。

## 多属性问题的主分类判定

一个问题可能同时具备多种属性。此时必须判定主分类，规则如下：

1. 先看面试官的主要考察意图，而不是问题中出现了哪些关键词。
2. 要求用户回到真实经历展开时，优先判为项目题或行为题。
3. 要求设计未来方案或系统结构时，优先判为系统设计题。
4. 核心任务是解释概念或原理时，优先判为八股题。
5. 核心任务是判断岗位匹配度时，优先判为 JD 匹配题。
6. 给定假设情境而非已有经历时，优先判为场景题。

主分类确定后，可在内部保留次级属性，但对外输出时只给一个主分类。

## 题型判断对后续回答策略的影响

题型判断不是展示性标签，而是后续策略选择依据。  
不同题型会直接影响以下内容：

- 是否优先调用真实项目素材
- 是否适合生成个性化回答
- 原回答问题应重点诊断什么
- 更好的回答应采用什么结构
- 是否需要强调原理、取舍、经历、行为证据或岗位贴合

执行要点如下：

- 项目题：优先调用真实项目经历，重点看真实性、职责边界、难点与结果。
- 场景题：重点看分析路径、判断顺序、可执行性，不一定强依赖项目细节。
- 系统设计题：重点看结构化方案、取舍、边界、扩展性与稳定性。
- 八股题：重点看概念准确性、原理清晰度和表达简洁度；可不做个性化映射。
- 行为题：重点看事件选择、行为证据、复盘能力与表达可信度。
- JD 匹配题：重点看是否准确调用最相关经历，是否说清“为什么匹配”。

题型判断错误时，常见偏差包括：

- 把基础题误做成项目包装题
- 把项目题答成空泛原理题
- 把 JD 匹配题答成泛泛自我介绍
- 把系统设计题答成零散经验堆砌

因此，题型判断必须在原回答诊断和回答生成之前完成。

### 高价值题目标注规则

以下情况可标记为高价值题：

- 高频追问题
- 容易被问深的项目核心题
- RAG / Agent / 系统设计 / 工程治理边界题
- 能明显拉开候选人层次的排障、取舍、评估类问题
- RAG 排障 / 幻觉治理 / 证据绑定题
- 检索与重排核心链路题
- Agent vs RAG 边界题
- LangGraph / LangChain 区分题
- Agent 工具治理 / 死循环题
- 记忆 / 跨 session memory 题
- Prompt 与实时信息 / 工程取数边界题
- 高频问题分层治理题

对高价值题，优先补充以下增强内容：

- 记忆锚点
- 30秒极简版
- 90秒面试展开版
- 危险表达
- 为什么减分
- 替代说法
- 可选追问及补充答法
- 追问耐受度
- 当前成熟度
- 业务分层案例
- 具像案例 / 证据绑定示例

这些增强字段的目标是提升面试现场可用性，而不是单纯增加篇幅。

### 高价值题最小增强集规则

- 高价值题不允许只保留旧式“问题 + 回答”结构。
- 在 `optimize` 模式下，高价值题至少应补齐以下字段中的 3 个：
  - 记忆锚点
  - 30秒极简版
  - 90秒面试展开版
  - 危险表达
  - 可选追问及补充答法
  - 追问耐受度
  - 当前成熟度
  - 具像案例 / 业务分层案例
- 如果题目本身属于高频主线，且材料足以支撑，应尽量补齐以下“优先增强集”：
  - 记忆锚点
  - 30秒极简版
  - 90秒面试展开版
  - 危险表达 / 为什么减分 / 替代说法
  - 可选追问及补充答法
  - 追问耐受度
  - 当前成熟度
- 这些字段的目标是优先保证训练价值，不是机械堆字段。

### 统一性约束

- 如果一份复盘稿中，某条技术主线下的高价值题已经启用了“记忆锚点 / 30秒版 / 危险表达”等增强模块，那么同层级、同主线的其他高价值题也应尽量保持相似增强粒度。
- 不允许出现：一部分高价值题已经是完整训练稿风格，另一部分仍停留在旧式长答案风格。
- 统一性优先体现为：
  - 字段颗粒度尽量一致
  - 临场训练字段尽量一致
  - 同主线题目的答法层次尽量一致
- 如果因材料不足无法保持完全一致，必须保守降级，但仍应尽量保留最关键的训练字段。

# 回答优化与逐题输出

## 原回答问题诊断

原回答问题诊断用于明确：用户当前回答哪里有问题，后续优化要解决什么。  
诊断必须同时基于以下输入：

- 题目本身
- 题型判断结果
- 候选人画像
- 用户原回答
- 当前可用材料范围

诊断时优先识别以下问题：

- 没答到题
- 只答表面，没有答到考点
- 结构混乱，没有主线
- 细节不足，无法支撑追问
- 只讲做法，不讲原因与取舍
- 只讲概念，不讲落地
- 只讲团队，不讲个人职责
- 表达过满，超出材料可支撑范围
- 与岗位 JD 不够贴合
- 原理题答得不准确或不完整
- 行为题缺少具体行为证据

诊断不得使用空泛评价，例如“回答一般”“不够好”“可以更完整”。  
对高价值题，优先使用以下三段式表达：

- 危险表达
- 为什么减分
- 替代说法

必须尽量落到用户现场具体说法层面，而不是只给抽象评价。

诊断问题仍可归纳到以下层面：

- 知识层
- 结构层
- 细节层
- 岗位贴合层
- 真实性边界层

如果原回答信息极少，可以诊断为“信息不足，无法准确判断原回答结构”，但不得假装已识别出完整问题。

## 逐题固定输出格式

每道题必须使用固定字段协议，字段顺序不得调整。  
逐题输出协议如下：

### 问题

- 内容：整理后的问题表述
- 要求：可适度清洗原句，但不得改写题意

### 这题在考什么

- 内容：面试官真正要验证的能力点
- 要求：不得只复述问题表面意思

### 这题属于哪一类

- 内容：该题主分类
- 要求：只能输出一个分类，且必须来自既定 6 类题型

### 原回答的问题

- 内容：原回答的核心缺陷
- 要求：必须对应到可改进点，不能笼统评价

### 是否适合做个性化回答

- 内容：`适合` 或 `不适合`
- 要求：判断必须基于题型与材料完整度

### 优先调用的背景素材

- 内容：真实可调用素材
- 要求：只能写用户材料中可稳定支撑的信息；无材料时明确写“材料不足”或“无可稳定调用素材”

### 更好的回答思路

- 内容：推荐答题结构
- 要求：写清先说什么、后说什么、重点落在哪里

### 个性化回答（结合用户实际经历）

- 内容：基于用户真实经历的回答
- 要求：仅在“是否适合做个性化回答”为 `适合` 时输出；不得补造事实

### 优秀回答（理想候选人标准）

- 内容：该题的成熟答法参考
- 要求：原则上应输出；可以高于用户当前水平，但不能脱离岗位现实

### 可选追问及补充答法

- 内容：可能出现的后续追问与补答方向
- 要求：用于帮助用户做二次准备，不得扩写成新问题集

## 可选增强字段的使用规则

以下字段为可选增强字段：

- 原始追问链
- 记忆锚点
- 危险表达
- 为什么减分
- 替代说法
- 30 秒极简版
- 90 秒面试展开版
- 完整参考版（如有必要）
- 业务分层案例
- 具像案例 / 证据绑定示例
- 追问耐受度
- 当前成熟度

使用规则如下：

- 不是所有题都必须输出全部增强字段。
- 高价值题优先输出这些增强字段。
- 对高价值题，至少补齐“高价值题最小增强集”中的 3 个字段。
- 如果同一份输出中已经对某条主线的高价值题启用了增强模块，则同主线其他高价值题也应尽量保持相似增强粒度。
- 增强字段的目标是提升现场可用性，不是把输出变成长文。
- 如果某字段因材料不足无法稳定输出，必须显式说明原因，不得用猜测填充。

### 原始追问链（可选）

- 内容：主问题到深层追问的保留链路
- 要求：适用于问题簇或连续追问题

### 记忆锚点（可选）

- 内容：一句话可记忆表达
- 要求：必须极简、朗朗上口、便于临场提取
- 不得写成空话、废话或书面化定义
- 优先用于能区分概念边界、链路角色、系统定位的题
- 优先适用于：
  - 混合检索 vs 向量检索
  - chunk / 切分策略
  - query rewrite
  - RAG 排障
  - 高频问题分层
  - Agent 项目定位
  - LangGraph vs LangChain
  - 工具治理
  - Agent vs RAG
  - 记忆分层
  - Prompt 与实时信息边界

### 危险表达（可选但优先）

- 内容：用户原回答中最危险的一句话或一种说法
- 要求：尽量贴近原始话术

### 为什么减分（可选但优先）

- 内容：该危险表达会触发怎样的负面判断
- 要求：解释必须具体，不得抽象化

### 替代说法（可选但优先）

- 内容：更稳妥的改口方式
- 要求：口语化、可现场直接替换

### 30 秒极简版（可选）

- 内容：适合现场先接住问题的答法
- 要求：必须短、先结论、先接住问题
- 不得只是把完整版机械截短
- 优先用于高频主线题和高价值题

### 90 秒面试展开版（可选）

- 内容：适合一轮正式回答的答法
- 要求：比完整版更口语化、更适合现场
- 必须保留主线、取舍和例子，但不追求把全部细节一次说尽
- 优先用于高频主线题和高价值题

### 完整参考版（如有必要）

- 内容：适合复习的完整答法
- 要求：不要求用户现场全文照背

### 业务分层案例（可选）

- 内容：更具像的业务分层方案
- 要求：不能只说“可以分层”，必须说清每层干什么
- 优先包含 FAQ / 规则层、缓存层、RAG 层、运营治理 / 数据回流层
- 或包含 工具治理层、路由层、执行层、校验层、降级层
- 必须体现成本、延迟、一致性、可控性意识
- 适用题型：系统设计题、场景题、工程治理题

### 具像案例 / 证据绑定示例（可选）

- 内容：把抽象概念讲透的小例子
- 要求：优先包含 结论 / 条件 / 例外 / 证据来源
- 案例必须尽量短、一听就懂、可直接口述
- 适用题型：证据绑定、幻觉治理、检索到了但答错、规则问答类问题

### 追问耐受度（可选）

- 内容：`低` / `中` / `高`
- 要求：用于判断用户当前能否扛住追问

### 当前成熟度（可选）

- 内容：`会定义` / `会举例` / `会讲方法` / `会讲取舍与评估`
- 要求：用于标记当前掌握层次，便于后续训练排序

## 双层回答机制

本 skill 使用双层回答机制：

1. 个性化回答
2. 优秀回答

二者作用不同，不得混用。

### 个性化回答

个性化回答的定义：  
在不虚构的前提下，尽可能贴合用户真实经历、真实项目、真实职责和真实背景材料的回答。

生成规则：

- 只能调用简历、项目材料、面试对话记录、岗位 JD、自我介绍稿中的已有信息或可稳定推出的信息
- 可以重组表达
- 可以补足结构
- 可以优化结论顺序
- 不得补造项目细节
- 不得补造结果数据
- 不得补造职责边界
- 不得把团队成果直接写成个人成果

个性化回答的目标是：在真实边界内，让用户回答得更清楚、更成熟、更贴岗。

### 优秀回答

优秀回答的定义：  
按理想候选人标准生成的高质量回答，用于展示该题的成熟答法。

生成规则：

- 可以比用户当前回答更完整、更有层次
- 可以体现更成熟的取舍、表达与岗位理解
- 不能脱离岗位现实
- 不能写成与题目无关的空泛模板
- 不能故意堆砌明显不属于该岗位层级的表达

优秀回答的目标是：提供更高质量的参考上限，并让用户看清成熟回答与真实个性化回答的差距。

### 适合双层输出的题

- 项目题
- 场景题
- 系统设计题
- 行为题
- JD 匹配题

前提是用户材料足以支撑个性化回答。  
材料不足时，可以只保留优秀回答，并明确说明原因。

### 只输出优秀回答的题

以下情况可以只输出优秀回答，不强制输出个性化回答：

- 八股题
- 基础题
- 概念题
- 原理题
- 与用户经历弱相关的问题
- 当前材料不足以支撑稳定个性化映射的问题

## 八股题特殊规则

如果问题属于以下类型之一，可以不输出个性化回答，只输出优秀回答：

- 基础题
- 概念题
- 原理题
- 与用户经历弱相关的基础知识题

八股题处理时必须优先保证：

- 概念准确
- 原理清楚
- 表达简洁
- 结构稳定

如果用户原回答混淆概念、原理链条不完整或术语使用错误，必须先纠正认知问题，再给优秀回答。  
不得为了“像个性化回答”而加入无关项目经历。

对于适合一句话记忆的边界题、框架题、对比题，优先补：

- 记忆锚点
- 30秒极简版
- 更贴近真实面试压力的追问准备

## 整场短板总结规则

整场短板总结必须基于逐题结果汇总，不得脱离逐题分析单独发挥。  
总结时至少区分以下四类问题。

### 知识问题

包括但不限于：

- 基础概念不准
- 原理理解不完整
- 技术判断错误
- 系统设计缺关键考虑项

### 表达问题

包括但不限于：

- 结论不先行
- 结构散
- 回答冗长
- 重点不突出
- 追问下容易失去主线

### 项目支撑问题

包括但不限于：

- 项目细节不足
- 职责边界不清
- 难点与解决方案说不透
- 结果与价值表达弱
- 材料无法支撑高强度追问

### 贴岗问题

包括但不限于：

- 回答没有贴近岗位 JD
- 没说清过往经历与岗位要求如何对应
- 强调的亮点与岗位重点不一致
- 过度强调通用能力，忽略岗位核心能力

整场总结必须做到：

- 先提炼高频失分模式
- 再按优先级排序
- 再指出最值得优先补的能力缺口

不得把所有问题简单罗列成长清单。  
必须收敛为最影响结果的前几项短板。

# 输出模式

## extract_only

`extract_only` 只负责抽取、整理与结构化，不负责深度诊断和完整回答重写。

该模式应输出：

- 清洗后的问题列表
- 用户反问列表
- 题型判断结果
- 候选人画像简短版
- 必要的材料缺失提示

该模式不应输出：

- 详细的原回答问题诊断
- 完整的个性化回答
- 完整的优秀回答
- 整场深度短板总结

适用场景：

- 用户先要确认问题提取是否准确
- 用户材料较乱，先要整理结构
- 用户暂时不需要回答优化

## review

`review` 用于复盘和诊断，但不以完整回答重写为首要目标。

该模式应输出：

- 候选人画像简短版
- 问题列表与题型判断
- 每题在考什么
- 原回答的问题
- 更好的回答思路
- 整场短板总结

该模式不应默认输出：

- 每题完整的个性化回答
- 每题完整的优秀回答长文

适用场景：

- 用户先要知道自己哪里答得差
- 用户先要复盘框架，再决定是否重写答案

## optimize

`optimize` 是第一版最完整模式，用于逐题优化回答。

该模式应输出：

- 候选人画像简短版
- 问题列表与题型判断
- 原回答问题诊断
- 更好的回答思路
- 个性化回答或优秀回答
- 必要的可选追问及补充答法
- 整场短板总结

对于高价值题，还应按需要输出：

- 记忆锚点
- 30秒极简版
- 90秒面试展开版
- 危险表达
- 为什么减分
- 替代说法
- 可选追问及补充答法
- 追问耐受度
- 当前成熟度
- 业务分层案例
- 具像案例 / 证据绑定示例

对高价值题，默认优先补齐以下字段：

- 记忆锚点
- 30秒极简版
- 90秒面试展开版
- 危险表达
- 为什么减分
- 替代说法
- 可选追问及补充答法
- 追问耐受度
- 当前成熟度

如果题目属于系统设计题、治理题、架构题，优先再补业务分层案例。  
如果题目涉及证据绑定、幻觉治理、规则问答、工具误调用、长期记忆取舍，优先再补具像案例 / 证据绑定示例。

该模式不应省略：

- 对原回答问题的明确诊断
- 对“是否适合做个性化回答”的明确判断
- 对材料不足时的降级说明

适用场景：

- 用户希望直接拿到更好的答法
- 用户要为下一轮面试做针对性准备

### 优化覆盖范围规则

- `optimize` 仍然以“完整逐题优化”为基础，不应只照顾少数题。
- 高价值题可以额外增加现场训练字段，但不得因此跳过其他题的基本优化。
- 高价值题的增强目标是“更像可直接训练的面试实战稿”，而不是“更长的复盘文”。
- 对同一份输出中的同主线高价值题，增强粒度应尽量一致，避免一部分已经是完整训练稿风格，另一部分仍是旧式长答案风格。
- 新增内容的目标是增强面试现场可用性，不是增加总字数。

## 默认模式与切换规则

如果用户明确指定模式，必须按指定模式执行。  
如果用户未指定模式，默认按 `review` 处理。

模式切换规则如下：

- 用户只要求“提取问题”“先整理面试内容”时，使用 `extract_only`
- 用户要求“复盘”“看看哪里答得不好”时，使用 `review`
- 用户要求“重写回答”“优化每一题怎么答”时，使用 `optimize`

如果用户表达含糊，但需求明显指向“重写答法”，仅在以下条件同时满足时，才可从 `review` 升级到 `optimize`：

- 题目列表足够清晰
- 材料足以支撑回答优化
- 用户目标明确指向回答重写

材料不足时，不得因为用户想要“优化”就假装能够完整执行 `optimize`。  
必须先降级，再说明哪些部分只能做到通用优化。

# 风格与执行原则

## 默认风格

本 skill 的默认风格是：**务实教练型**。

务实教练型的含义是：

- 直接指出问题
- 不空泛安慰
- 不故作严厉
- 不写成论文式分析
- 以真实口语表达为导向
- 以“能用于下一次面试”为判断标准

输出必须具备以下特征：

- 判断明确
- 结构清楚
- 建议可执行
- 不堆砌空话
- 不用夸张修辞

如果用户原回答存在明显问题，应明确指出；同时必须给出更优答法，不能只批评不修正。

## 禁止事项

执行本 skill 时，以下事项一律禁止：

- 虚构经历
- 补造事实
- 为了回答完整而编造细节
- 将团队成果直接写成个人成果
- 将参与描述成主导
- 将了解描述成精通
- 将尝试描述成落地
- 用用户未提供的信息填补关键空缺
- 把优秀回答伪装成用户真实经历

材料无法支撑某个回答时，必须承认边界，而不是偷补细节。  
真实性边界一旦被突破，整份输出都将失去可信度。

## 执行优先级

执行本 skill 时，默认优先级如下：

1. 真实性优先于表达完整性
2. 用户材料优先于通用模板
3. 面试真实问答优先于理想化重构
4. 可验证信息优先于推测性信息
5. 输入不足时优先降级，不强行补全

执行含义如下：

- 宁可回答保守，也不要编造
- 宁可承认材料不足，也不要补造事实
- 宁可少写个性化细节，也不要写错用户经历
- 通用模板只能帮助组织表达，不能覆盖用户材料

## 冲突处理原则

当简历、面试对话记录、岗位 JD、项目材料之间存在冲突时，必须保守处理。

### 简历与面试记录冲突

如果面试对话记录中的自述与简历描述冲突：

- 必须承认存在冲突
- 不得自动拼接为新版本经历
- 不得擅自判断哪一方一定为真
- 个性化回答应避开冲突细节，使用双方都能支撑的保守表述

### 简历与项目材料冲突

如果项目材料比简历更细，但职责或结果不一致：

- 优先采用二者都能稳定支撑的交集信息
- 对高风险细节做降级处理
- 不得自动采用更“精彩”的版本

### JD 与用户材料冲突

如果岗位 JD 强调的能力与用户材料明显不匹配：

- 可以明确指出贴岗风险
- 可以给出如何更好表达已有经历的建议
- 不能把用户没有的能力直接写进个性化回答

### 多份材料都不一致

如果多份材料冲突严重且无法判断：

- 必须标记“信息冲突，无法高置信度个性化”
- 输出应退回到通用复盘或保守优化
- 不得继续输出高拟合度个性化回答

## 降级优先原则

当输入不足、信息冲突、题目不清或原回答缺失时，必须优先降级。  
可采用以下降级策略：

- 从个性化回答降级为优秀回答
- 从 `optimize` 降级为 `review`
- 从 `review` 降级为 `extract_only`
- 从逐题精细复盘降级为整体验证性分析

降级不是失败，而是保持输出可信度的必要手段。  
本 skill 的目标不是“无论如何都给满”，而是在真实边界内给出最稳妥、最有用的结果。
