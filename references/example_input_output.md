## 6. 一个“适合个性化回答”的完整逐题示例

```md
问题
你怎么让回答绑定证据，而不是检索到了资料却还是答错？

这题在考什么
在考你是否真正理解“检索到了资料”不等于“回答一定正确”，以及你能不能把证据绑定、结构约束和业务规则表达清楚。

这题属于哪一类
场景题

原始追问链
- 如果检索出来的内容不准，或者模型开始胡说，你一般怎么排查？
- 如果 prompt 已经写了不要幻觉，为什么还是会幻觉？
- 你怎么让回答绑定证据，而不是检索到了资料却还是答错？
- 深层考察意图：是否真正理解幻觉治理和证据绑定不是一句“调 prompt”就能解决

原回答的问题
- 危险表达：我理解就是在 prompt 里面强调参考资料就行。
- 为什么减分：这句话会让面试官判断你把证据绑定理解成单点 prompt 技巧，没有体现结构约束、证据选择、业务规则表达和结果校验。
- 替代说法：我不会只靠一句“请参考资料回答”，而是会同时做证据筛选、答案结构约束和结果校验，确保模型不是“看过资料但还是答偏”。

是否适合做个性化回答
适合

优先调用的背景素材
- 做过知识库问答助手的检索链路
- 做过文档切分优化
- 做过简单评估表，关注命中率、引用正确性、明显幻觉
- 做过规则问答的结构约束优化

更好的回答思路
先讲核心结论，再讲“为什么只写 prompt 不够”，然后用一个业务小例子把证据绑定说透，最后补验证方式。

记忆锚点
证据绑定不是“让模型看资料”，而是“让答案的每个判断都能对上证据”。

30秒极简版
我不会把证据绑定理解成一句“请参考资料回答”。真正有效的是三层一起做：先保证召回片段是对的，再限制回答结构必须写结论、条件、例外和证据来源，最后再做结果校验。这样可以减少“检索到了资料但还是答错”的情况。

90秒面试展开版
我理解证据绑定不是让模型“看过资料”就结束了，而是要让答案里的关键判断都能对应到具体证据。实际做法上我一般会分三步。第一步先看召回片段是不是相关且完整，如果片段本身不完整，后面再怎么约束都容易答偏。第二步在生成阶段加结构约束，比如要求答案按“结论、条件、例外、证据来源”输出，而不是让模型自由发挥。第三步做简单校验，比如检查答案里的关键结论有没有对应到正确资料，避免它拿着相近片段答错规则。这样做比单纯写一句“不要幻觉”要稳很多。

业务分层案例
- FAQ / 规则层：高频、结构稳定、规则明确的问题优先直接命中 FAQ 或规则模板，成本低、延迟低、稳定性高。
- 缓存层：对高频重复问法做结果缓存或标准问法归并，减少重复检索和生成开销。
- RAG 层：对需要结合制度文档、产品说明的复杂问题再走检索和生成，重点控制召回质量和证据绑定。
- 运营治理 / 数据回流层：记录高风险问题、错误答案和用户追问，回流做 FAQ 补充、chunk 调整和评估集更新。

具像案例 / 证据绑定示例
- 结论：退款申请提交后 24 小时内可以撤回。
- 条件：订单状态必须还在“审核中”。
- 例外：如果已经进入打款流程，则不能撤回。
- 证据来源：chunk1=退款规则总则，chunk2=审核状态说明，chunk3=打款流程限制。

个性化回答（结合用户实际经历）
我不会把证据绑定理解成一句“请参考资料回答”就够了，我更倾向于把它拆成“召回质量 + 结构约束 + 结果校验”三层。  
在我做的企业知识库问答助手里，我们面对的很多是制度和规则类问题，这类问题如果只把检索结果丢给模型，它可能看到了相关资料，但最后还是把条件和例外说错。后来我在回答阶段加过比较强的结构约束，比如要求按“结论、条件、例外、证据来源”输出，而不是自由组织答案。  
这样做的好处是，模型不只是“参考资料”，而是要把关键判断落到具体证据上。再结合简单评估去看命中率、引用正确性和明显幻觉，能更容易发现是召回片段不完整，还是答案组织过程出了问题。

优秀回答（理想候选人标准）
我会把“证据绑定”理解成：答案中的关键判断必须能追溯到具体证据，而不是模型只要“看过资料”就算完成。  
实际做法上我会控制三层。第一层是检索层，确保召回片段相关且完整；如果 chunk 本身切得太碎，模型看到的证据就不完整，后面很容易答偏。第二层是生成层，要求答案按固定结构输出，比如“结论、条件、例外、证据来源”，这样模型不容易把规则问答说成泛化总结。第三层是校验层，检查最终答案里的关键结论是否真的对应到检索证据，而不是只引用了相近但不支持结论的内容。  
如果是业务上更稳的做法，我还会分层处理：高频 FAQ 和明确规则优先走 FAQ / 规则层，复杂问题再走 RAG，最后再把错误案例回流到运营治理层做补充。这样在成本、延迟和稳定性上都更可控。

可选追问及补充答法
- 追问：如果 prompt 已经写了不要幻觉，为什么还是会幻觉？
  - 补答方向：说明 Prompt 只能约束生成，不解决证据不完整、召回错片段、规则结构缺失这些问题。
- 追问：如果 chunk 检索到了，但结论还是错，最可能错在哪？
  - 补答方向：可能是检索到了相近证据但不支持最终结论，也可能是模型在答案组织时把条件和例外合并错了。
- 追问：为什么很多规则问答先走 FAQ 层，再走 RAG？
  - 补答方向：因为规则稳定、成本更低、延迟更小，且更容易保证一致性。
```

## 7. 一个“Agent 工具治理 / 死循环题”的完整逐题示例

```md
问题
如果 Agent 调工具时一直重复调用、像死循环一样转不出来，你怎么治理？

原始追问链
- 你怎么判断这是模型能力问题，还是治理链路问题？
- 如果你已经加了“最多调用 3 次”的限制，但它还是在错误路径上反复试怎么办？
- 深层考察意图：判断你是否真正理解 Agent 失败闭环、工具治理和降级机制

这题在考什么
在考你是否能把 Agent 死循环问题拆到路由、工具选择、执行校验、状态管理和降级策略几个层面，而不是只会说“限制调用次数”。

这题属于哪一类
场景题

原回答的问题
- 危险表达：我一般就给它设个最大轮数，超过就停掉。
- 为什么减分：这会让面试官觉得你只做了最表层的保护，没有真正治理“为什么会走错、为什么会反复走错、停掉之后怎么收口”。
- 替代说法：我会把它拆成“为什么会重复调用”和“重复后怎么收口”两层，前者看路由和校验，后者看降级和人工接管。

是否适合做个性化回答
不适合。当前示例材料里没有稳定的 Agent 项目经历，适合输出优秀回答和训练版话术，不强行绑定个人项目。

优先调用的背景素材
无可稳定调用素材。当前更适合作为高价值题训练模板使用。

更好的回答思路
先给结论：死循环不是单点问题，而是治理链路没收口。再按“为什么会循环 -> 怎么拦截 -> 拦截后怎么降级”展开。

记忆锚点
Agent 死循环不是“模型太笨”，而是“治理链路没收口”。

30秒极简版
我会把 Agent 死循环看成治理问题，不只是在外面加一个最大轮数。核心要做三件事：先限制错误路径继续放大，再校验每次工具调用是不是合理，最后给出明确降级出口，比如转人工、转 FAQ、或者返回澄清问题。

90秒面试展开版
如果 Agent 一直重复调工具，我不会只看成“模型输出不稳定”，而会先把问题拆成三层。第一层是为什么会循环，比如路由判断错了、工具描述太模糊、状态没有更新，导致它以为自己还没完成任务。第二层是怎么拦截，我会在执行前后都加校验，比如参数是否合法、结果是否真的推进了状态、同一个工具是否被无意义重复调用。第三层是怎么收口，如果多轮之后仍然没有有效推进，就不能继续让它试，而是要明确降级，比如转 FAQ、让用户补信息，或者进入人工处理。这样面试里我想表达的是，Agent 死循环不是单一 prompt 问题，而是路由、执行、校验、降级几层要一起治理。

业务分层案例
- 工具治理层：定义每个工具能做什么、不能做什么、输入输出约束是什么
- 路由层：判断当前任务到底该不该调工具、该调哪个工具
- 执行层：真正发起调用，并记录调用结果和状态变化
- 校验层：检查这次调用有没有推进任务，还是只是重复动作
- 降级层：超过阈值后转 FAQ、澄清提问或人工接管，避免无限尝试

优秀回答（理想候选人标准）
我会把 Agent 死循环看成一个典型的“执行闭环失控”问题，而不是简单理解成模型多试了几次。治理上我通常会分五层看。第一层是工具定义是否清楚，如果工具边界和返回结构不清楚，模型很容易误判。第二层是路由是否合理，避免本来不该调工具的问题也被送去执行。第三层是执行后有没有状态推进，如果调用结果没有改变状态，系统就会不断重复原动作。第四层是校验机制，比如参数校验、结果校验、重复调用检测和失败原因分类。第五层是降级出口，如果连续几轮没有有效推进，就必须切到 FAQ、澄清问题或人工处理，而不是继续试。  
如果现场要强调工程意识，我会再补一句：治理目标不是一味追求“让 Agent 自己想明白”，而是让系统在成本、延迟和可控性之间更稳地收口。

可选追问及补充答法
- 追问：如果 prompt 已经写了“不要重复调用”，为什么还是会重复？
  - 补答方向：因为 prompt 只能影响倾向，不能替代状态更新、结果校验和错误收口机制。
- 追问：你说加校验，那如果校验本身也误判怎么办？
  - 补答方向：校验不要只靠单规则，至少结合参数合法性、状态推进和结果结构三个维度判断。
- 追问：如果限制太严，会不会把该成功的复杂任务也拦掉？
  - 补答方向：所以要区分“合理重试”和“无效重复”，并把复杂任务放到可控的多阶段执行链里。

追问耐受度
中

当前成熟度
会讲方法
```

## 8. 一个“Prompt 与实时信息边界题”的完整逐题示例

```md
问题
Prompt 能不能解决实时信息问题？比如今天库存、实时价格、工单状态这类信息，能不能靠 prompt 写清楚就行？

这题在考什么
在考你是否真正理解 Prompt 的作用边界，以及你能不能把“表达约束”和“实时取数链路”区分开。

这题属于哪一类
场景题

原回答的问题
- 危险表达：可以，prompt 写细一点，让模型先别乱答就行。
- 为什么减分：这会让面试官觉得你把“怎么说”误当成“信息从哪里来”，没有建立实时数据必须靠外部系统提供的基本边界。
- 替代说法：Prompt 可以约束回答方式，但实时信息要靠外部数据源、接口或工具调用提供，prompt 解决不了“数据本身不存在”这个问题。

是否适合做个性化回答
不适合。当前示例材料里没有稳定的实时数据项目经历，更适合作为边界题训练模板使用。

优先调用的背景素材
无可稳定调用素材。当前重点是把边界讲清楚，而不是强行绑定经历。

更好的回答思路
先一句话划边界，再补一个实时库存或价格的小例子，最后说明 prompt 和实时链路各自负责什么。

记忆锚点
Prompt 决定“怎么说”，实时链路决定“能不能说”。

30秒极简版
我不会把实时信息问题交给 prompt 解决。Prompt 能做的是约束回答格式和行为边界，但库存、价格、工单状态这类信息必须从外部系统实时取，拿不到数据，再好的 prompt 也只能让模型更谨慎地胡说。

90秒面试展开版
我会把这个问题分成两层。第一层是 prompt 的作用，它可以规定模型怎么表达，比如要求引用来源、拿不到信息就不要编、先返回澄清语句。第二层是实时信息链路，它决定系统有没有能力拿到今天的库存、当前价格或者最新工单状态。像库存和价格这种信息本身在持续变化，模型参数和静态 prompt 都不会自动知道最新值，所以必须通过接口、数据库查询或者工具调用把数据取回来。换句话说，prompt 解决的是“回答策略”，实时链路解决的是“数据事实”。如果没有后者，前者最多只能让系统更保守，不能让答案真正变成实时正确。

优秀回答（理想候选人标准）
Prompt 不能替代实时取数链路，它只能约束模型在拿到信息之后如何表达。  
如果问题本身依赖动态数据，比如今天库存、当前价格、工单最新状态，系统必须通过 API、数据库或工具调用获取最新事实，然后再由模型把结果组织成自然语言。Prompt 可以补充的，是让模型在没有数据时明确说“不知道”、在有数据时按固定结构回答、或者要求引用时间戳和来源。  
所以工程上要分清两件事：一是“数据从哪里来”，二是“拿到数据后怎么说”。前者是实时链路设计，后者才是 prompt 设计。

可选追问及补充答法
- 追问：那如果我把最新库存每天都塞进 prompt，不也算实时吗？
  - 补答方向：那本质上还是外部数据在更新，只是更新方式很脆弱，无法保证时效性、一致性和维护成本。
- 追问：如果接口超时了，你会怎么处理？
  - 补答方向：要有降级策略，比如返回“当前无法获取最新状态”，而不是让模型猜一个结果。
```
